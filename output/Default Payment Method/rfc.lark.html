Glossary

Term

Definition

GPL

GoPay Later (the pay-later product this RFC concerns).

Default Payment Method

A per-user, per-platform flag indicating GPL should be the pre-selected payment instrument on Gojek / GoPay checkout surfaces.

Virgin user

A GPL-whitelisted user who has not yet submitted / been approved for a credit application.

Post-credit flow

The side-effect chain that runs after a credit-application-success callback is received by lending-adapter.

paylater-user-service

Java 21 / Spring Boot 3.5 service owning per-user GPL data (DDL module gtfpaylater). DDD layout: shared/infrastructure/domain/application/web/scheduler.

gpl-bff

Kotlin / Spring Boot 3.1 WebFlux BFF. Reactive Feign clients. Owns client-facing orchestration only.

lending-adapter

Kotlin / Spring Boot 2.7 WebFlux adapter that receives the upstream lender (BizEnd/MAB) credit-application callback and runs the post-credit flow.

DEFAULT_PAYMENT_OPTION

Existing enum value in gpl-bff UserSettingFeature and a disabled entry in toggle-configs.json — scaffolding this RFC activates.

Background

GoPay Later is live in QRIS (CPM since Sep 2025; MPM ON-US Jan; OFF-US Feb). As of April there are 485k eligible users but only 11% adopted. The PRD wants to grow adoption by letting users set GPL as their default payment method across three flows: acquisition, transaction, and account-management (settings).

This RFC is scoped to the backend work for two of those three flows — the credit-application / post-credit flow and the setting page — and explicitly excludes the transaction nudge flow (PRD §B, US5–US9). The transaction nudge is tracked separately and will consume the read API defined here.

Key ownership decision (per engineering context): the GPL default-payment-method setting is owned by paylater-user-service. We will not call GoPay APIs to store or read this setting; GoPay-side checkout propagation is handled by a separate integration (see Open Questions).

Desired State

  • paylater-user-service owns a new user_default_payment_setting table (MySQL, gtfpaylater datasource) keyed by gofin/gopay user id, storing is_default_enabled, opt_in_source (ONBOARDING_REVIEW / POST_APPROVAL / SETTINGS_PAGE), treatment_group, and audit timestamps.
  • Two new internal endpoints on paylater-user-service: a read (GET) and an idempotent write/apply (POST) for the default-payment-method setting, under /user-service/internal/user/default-payment-method.
  • gpl-bff GET /account-management/settings/toggles additionally fetches the default-payment-method status from paylater-user-service and surfaces the (existing) DEFAULT_PAYMENT_OPTION toggle with show:true behind a feature flag.
  • gpl-bff PATCH /account-management/settings/toggle for DEFAULT_PAYMENT_OPTION is redirected from bizend to paylater-user-service.
  • gpl-bff exposes a new internal endpoint POST /internal/credit-application/default-payment-method/apply (called by lending-adapter on approval) that forwards the apply request to paylater-user-service.
  • lending-adapter post-credit flow gains a 4th parallel side-effect: call gpl-bff POST /internal/credit-application/default-payment-method/apply so the cached onboarding opt-in is applied on success. Idempotent.
  • Experiment modulo logic (US2) lives in paylater-user-service with a config-driven rollout percentage and a global kill-switch.

In Scope

  • New user_default_payment_setting table + PO + mapper + domain model in paylater-user-service.
  • Read & write/apply internal APIs on paylater-user-service.
  • Experiment modulo + rollout config + kill-switch in paylater-user-service.
  • BFF toggle read enrichment + write redirection for DEFAULT_PAYMENT_OPTION.
  • BFF internal apply endpoint consumed by lending-adapter.
  • lending-adapter post-credit side-effect to apply the cached default on approval.
  • Feature flag, observability, idempotency for the new apply path.

Out of Scope

  • Transaction nudge flow (PRD US5–US9): cache-on-confirm, risk-reject handling, PN fallback, backward-compat gating. Separate RFC. Will reuse the read/write API defined here.
  • GoPay checkout widget propagation: how GPL actually becomes the pre-selected instrument on the GoPay payment widget (US4 "Default label" at checkout) is a GoPay-side integration, tracked under Open Question OQ-2.
  • Frontend / client work: acquisition review page UI, post-approval screen UI, settings screen UI.
  • Tokopedia default-payment-method (explicitly out of scope in the PRD).
  • Data migration of any pre-existing default setting (none exists).

Solution

Recommended approach: route every read/write of the default-payment-method setting through paylater-user-service, and trigger the post-credit apply via the existing lending-adapter → gpl-bff call chain.

Rationale: (1) respects the ownership boundary set by engineering — the GPL user setting lives in the GPL domain, not in GoPay or bizend; (2) reuses the existing gpl-bff → paylater-user-service Reactive Feign client (GplPaylaterUserServiceClient.kt:9-19) and the existing lending-adapter → gpl-bff callback path (GopayLaterBffClient.kt:53-92), so no new cross-service network topology; (3) the BFF stays a thin orchestrator, matching the ToggleSettingService.fetchToggles aggregation pattern already in place (ToggleSettingService.kt:58-89).

Repository ownership (this RFC):

  • ~/project/paylater-user-service/ — source of truth for the default-payment-method setting; experiment modulo logic; read + apply APIs.
  • ~/project/gpl-bff/ — client-facing /account-management/settings entry point (read enrichment + write redirection); internal apply endpoint for lending-adapter.
  • ~/project/lending-adapter/ — credit-application-success callback; new post-credit side-effect that calls gpl-bff to apply the cached default.

What we will NOT do: add a lending-adapter → paylater-user-service direct client (considered — one fewer hop — but rejected: it creates a new cross-service dependency not present today, bypasses the BFF orchestration boundary, and diverges apply from read/write into two integration paths); call GoPay to store the setting; or backfill via bizend.

Technical Approach

Overview

lending-adapter PKYCCreditApplicationService.handleCreditApplicationCallback gains a 4th branch in the existing Mono.zip at PKYCCreditApplicationService.kt:792: call gplBffClient.applyDefaultPaymentMethod(...) → POST /internal/credit-application/default-payment-method/apply. gpl-bff adds that internal endpoint (InternalController.kt:31), which forwards to paylater-user-service POST /user-service/internal/user/default-payment-method/apply.

Idempotency: the apply endpoint sets is_applied=true only if false (conditional UPDATE), so duplicate callbacks are safe. The post-approval-screen write (AC 3.2) reuses the same write API as US1 with optInSource=POST_APPROVAL and forces is_applied=true (the user is already approved).

  4. Phase 3 (prod, 100%): Rollout-pct = 100 after experiment concludes. Remove kill-switch only after sign-off.

Monitoring gate per phase: apply failure rate < 0.1%, PUS read p99 < 200ms, no increase in post-credit callback latency beyond existing budget.
